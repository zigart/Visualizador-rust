## Context

Tras `persistence-04` y `usuario-viajes-07`, las tablas tienen tanto columnas del contrato de movimientos (`id_recorrido`, `id_usuario`, `operacion`, `fechahora`, `id_estacion`) como columnas legacy del bootstrap inicial. El INSERT actual en `recorrido.rs` escribe 10 columnas; el contrato objetivo requiere 6 columnas de negocio mas `id` autogenerado.

`estado_bicicletas` fue renombrada a `en_uso`/`maximo_historico` en migracion `003`, pero conserva `SMALLINT PRIMARY KEY`, constraint singleton y `updated_at`.

## Goals / Non-Goals

**Goals:**

- Dejar `recorridos` y `estado_bicicletas` con el esquema objetivo documentado en la spec.
- Migracion incremental segura para entornos de desarrollo (puede truncar o migrar datos legacy segun estrategia elegida).
- Adaptadores PostgreSQL alineados: INSERT/UPDATE/SELECT solo sobre columnas del contrato.

**Non-Goals:**

- No cambiar reglas de dominio ni contrato RabbitMQ.
- No modificar API HTTP ni WebSocket.
- No sincronizar automaticamente con migraciones del servicio TypeScript (solo alinear contrato descrito).

## Decisions

### Migracion por recreacion de tablas

Para eliminar multiples columnas `NOT NULL` legacy y cambiar tipos (`BIGSERIAL` â†’ `SERIAL`, `BIGINT` â†’ `INTEGER`), la migracion:

1. Crea tablas nuevas `recorridos_new` y `estado_bicicletas_new` con esquema objetivo.
2. Copia datos mapeando solo columnas comunes (`id_recorrido`, `id_usuario`, `operacion`, `fechahora`, `id_estacion`, `en_uso`, `maximo_historico`).
3. Renombra tablas viejas a `_legacy` y nuevas al nombre final.
4. Recrea indices utiles (`recorridos_usuario_fechahora_idx`, `recorridos_id_recorrido_operacion_idx`).

**Alternativa considerada**: `ALTER TABLE DROP COLUMN` en cascada. Descartada porque mezcla tipos incompatibles y deja indices huÃ©rfanos.

### Tipos PostgreSQL vs contrato

| Contrato objetivo | PostgreSQL |
|-------------------|------------|
| SERIAL | `SERIAL` / `INTEGER GENERATED BY DEFAULT AS IDENTITY` |
| INTEGER | `INTEGER` |
| VARCHAR(20) | `VARCHAR(20)` con CHECK opcional `operacion IN ('retiro','devolucion')` |
| TIMESTAMPTZ | `TIMESTAMPTZ` |
| BIGINT NULL | `BIGINT` nullable |

`id_estacion` se mantiene como extension Rust (`BIGINT NULL`).

### estado_bicicletas sin singleton

El contrato objetivo usa `id SERIAL PRIMARY KEY` sin constraint `id = 1`. La aplicacion seguira leyendo/actualizando la fila con `id = 1` (seed en migracion) o la unica fila existente via `ORDER BY id LIMIT 1` si hay una sola fila.

**Alternativa considerada**: mantener singleton. Descartada por requisito explicito de eliminar CHECK singleton.

### INSERT simplificado en recorrido.rs

```sql
INSERT INTO recorridos (id_recorrido, id_usuario, operacion, fechahora, id_estacion)
VALUES ($1, $2, $3, $4, $5)
```

### UPDATE simplificado en estado.rs

```sql
UPDATE estado_bicicletas
SET en_uso = $1, maximo_historico = $2
WHERE id = 1
```

## Risks / Trade-offs

- [Perdida de datos legacy no mapeables] â†’ Solo se conservan columnas del contrato; columnas bootstrap se descartan.
- [Entornos con migraciones ya aplicadas] â†’ Requiere `sqlx migrate run`; documentar en README.
- [Tests testcontainers] â†’ Deben validar esquema post-migracion, no columnas eliminadas.

## Migration Plan

1. Aplicar migracion en desarrollo con `docker compose up` + `sqlx migrate run`.
2. Verificar `cargo test -p adaptadores --test persistencia_postgres`.
3. Smoke: `cargo run -p api`, publicar retiro/devolucion, consultar `/usuarios/:id`.

Rollback: restaurar backup de PostgreSQL o revertir migracion manualmente (no hay down automatico en primer incremento).

Seed de `estado_bicicletas` siempre `id=1`
`operacion IN ('retiro','devolucion')` validar en dominio
